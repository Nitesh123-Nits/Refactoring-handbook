Introduction to Code Refactoring

What is Refactoring?

Refactoring is the disciplined process of restructuring existing code — changing its internal structure without altering its external behavior. The term was popularized by Martin Fowler in his seminal book Refactoring: Improving the Design of Existing Code (1999), but the practice predates it by decades.

The key word is disciplined. Ad-hoc cleanup driven by intuition is not refactoring — it's hacking. True refactoring follows a systematic approach: identify a problem, apply a named technique, verify behavior is preserved, commit. Repeat.

💡 Fowler's Definition
Refactoring is a change made to the internal structure of software to make it easier to understand and cheaper to modify, without changing its observable behavior.

Why It Matters in Production Systems

Technical debt compounds. A 2023 study by McKinsey estimated that engineering teams spend 20–40% of their time managing technical debt. In large enterprise systems, that number climbs higher. The cost is not just developer time — it's:

  • Velocity loss: New features take 3x longer because every change touches fragile, tangled code.
  • Bug rate: Poorly structured code correlates directly with defect density.
  • Onboarding time: New engineers spend weeks understanding spaghetti code instead of shipping.
  • Incident rate: Complex, tightly-coupled systems are harder to diagnose and fix under pressure.

Refactoring vs Rewriting

DimensionRefactoringRewriting
RiskLow — incremental, testedExtremely high — "big bang"
TimeWeeks to months, parallel to featuresMonths to years, often blocks features
Knowledge preservationHigh — business rules stay intactLow — edge cases get lost

⚠ The Rewrite Trap
Joel Spolsky famously called rewriting from scratch "the single worst strategic mistake a software company can make." Netscape 6 rewrote the browser from scratch — it took 3 years, was riddled with bugs, and cost Netscape its market lead. Incremental refactoring almost always wins.

✅ After — Clean, Single Responsibility
@Service
@Transactional
public class OrderService {

    private final OrderValidator validator;
    private final InventoryService inventory;
    private final PricingEngine pricing;
    private final NotificationService notifications;
    private final OrderRepository orderRepo;

    public OrderResult processOrder(CreateOrderRequest req) {
        validator.validate(req);

        List<OrderItem> items = req.items().stream()
            .map(this::resolveItem)
            .toList();

        inventory.reserve(items);  // throws if unavailable

        Money total = pricing.calculate(items);
        Order order = orderRepo.save(Order.create(req, items, total));

        notifications.sendConfirmation(order);
        return OrderResult.success(order.getId(), total);
    }

    private OrderItem resolveItem(ItemRequest r) {
        return inventory.findBySku(r.sku())
            .toOrderItem(r.quantity());
    }
}

Core Principles

Clean Code Principles

Robert C. Martin's Clean Code distills decades of software craftsmanship into actionable heuristics. These aren't aesthetics — they directly impact defect rates and maintainability.

PrincipleCore IdeaCommon Violation
Meaningful NamesNames should reveal intentint d; // elapsed time
Small FunctionsFunctions do ONE thing200-line methods with multiple levels of abstraction
No Side EffectsFunctions do what their name says, nothing moreA getUser() that also logs audit events
Prefer ExceptionsDon't return error codesReturning null or -1 on failure

SOLID Principles

S — Single Responsibility Principle

A class should have only one reason to change. If your UserService handles authentication, email sending, and PDF generation — it has three reasons to change. Each is a separate responsibility that should live in its own class.

O — Open/Closed Principle

Classes should be open for extension, closed for modification. Add new behavior by adding new code, not by changing existing code. Strategy and Template Method patterns are the canonical implementations.

L — Liskov Substitution Principle

Subtypes must be substitutable for their base types without altering the correctness of the program. If your Square extends Rectangle breaks when you call setWidth(), you're violating LSP.

I — Interface Segregation Principle

Clients should not be forced to depend on interfaces they don't use. A 20-method UserRepository interface is almost certainly wrong — split it into focused ports.

D — Dependency Inversion Principle

High-level modules should not depend on low-level modules. Both should depend on abstractions. This is the bedrock of testability — if your service instantiates its own database connection, you cannot unit test it.

❌ Violates DIP
public class ReportService {
    // Hardwired dependency — untestable
    private MySQLReportRepository repo =
        new MySQLReportRepository();

    public Report generate(Long id) {
        return repo.findById(id);
    }
}
✅ DIP via Constructor Injection
public class ReportService {
    // Depends on abstraction, not implementation
    private final ReportRepository repo;

    public ReportService(ReportRepository repo) {
        this.repo = repo;
    }

    public Report generate(Long id) {
        return repo.findById(id)
            .orElseThrow(() -> new ReportNotFoundException(id));
    }
}

DRY, KISS, YAGNI

🔁
DRY — Don't Repeat Yourself
Every piece of knowledge should have a single, authoritative representation in the system. Duplication is the root of most maintenance pain — when the business rule changes, you update it in one place or you don't.
HIGH PRIORITY
💡
KISS — Keep It Simple, Stupid
The simplest solution that works is almost always the right one. Resist the urge to build a generic, extensible framework when a straightforward class will do. Complexity is a liability, not an asset.
MEDIUM PRIORITY
🚫
YAGNI — You Ain't Gonna Need It
Don't build features or abstractions based on speculation about future requirements. Over-engineering is as harmful as under-engineering — you pay the complexity cost now, possibly forever, for value that never materializes.
MEDIUM PRIORITY

Identifying Code Smells

Code smells are surface indicators of deeper problems in the design. The term was coined by Kent Beck and popularized by Martin Fowler. Smells don't always mean a problem — they're signals that warrant investigation.

The Smell Catalog

📏
Long Method
Methods longer than 20–30 lines are hard to understand, test, and reuse. If you need to scroll to read a method, it's doing too much. Extract into smaller, named methods.
HIGH
🦄
God Class
A class that knows too much or does too much. Often has 50+ methods, thousands of lines. Everything depends on it, making changes terrifying. The classic monolith in class form.
HIGH
📋
Duplicate Code
The same code structure appearing in more than one place. When the duplicated logic needs to change, you have to find every copy. Guaranteed to diverge over time.
HIGH
🔗
Tight Coupling
Classes that directly instantiate each other or share mutable state. Changes ripple unpredictably. Impossible to test in isolation. The enemy of modular design.
HIGH
📦
Primitive Obsession
Using primitives (String, int) where domain objects should exist. A userId is not just a Long — it's a concept with behavior. An email address is not just a String — it has validation rules.
MEDIUM
⛓
Long Parameter List
Methods with more than 3–4 parameters are hard to call correctly and understand. Replace with a Parameter Object or Builder. A method accepting 8 booleans is a design crime.
MEDIUM
🔀
Divergent Change
One class changed for many different reasons. If you find yourself saying "I changed X because of pricing AND because of tax AND because of discounts," the class has too many responsibilities.
MEDIUM
🌊
Shotgun Surgery
Opposite of divergent change — one change requires modifying many classes. If adding a new payment method requires touching 12 files, your design needs consolidation.
HIGH
🤝
Feature Envy
A method that seems more interested in data from another class than its own. A method in OrderService that accesses 8 fields of Customer directly is probably a method that should live in Customer.
MEDIUM
🎭
Switch Statements
Large switch or if-else chains on type codes usually indicate missing polymorphism. Every time a new type is added, every switch statement must be updated — a classic OCP violation.
MEDIUM
🏚
Dead Code
Unreachable code, unused variables, commented-out blocks. Increases cognitive load, causes confusion about intent, and may be hiding bugs. Delete it — version control is your undo button.
LOW
💬
Comments as Deodorant
When you need a comment to explain WHAT code does (not WHY), the code is unclear. The comment is masking a naming or structural problem. Extract and rename instead of commenting.
LOW

Refactoring Techniques (LLD Focus)

Extract Method

Turn a code fragment into a method whose name explains the purpose of the fragment.

FOWLER CATALOG #1

Problem

A method contains a block of code that can be grouped together. The block is complex enough that it needs a comment to explain it.

Steps

  1. Identify the fragment you want to extract
  2. Create a new method named after what it does (verb-noun: calculateDiscount, validateEmail)
  3. Copy the fragment into the new method
  4. Scan for local variables — pass them as parameters or return them
  5. Replace the original fragment with a call to the new method
  6. Run tests — behavior must be identical
❌ Before
void printOrder(Order order) {
    System.out.println("Order #" + order.getId());
    System.out.println("Customer: " + order.getCustomer().getName());
    // calculate total with tax
    double subtotal = 0;
    for (OrderItem i : order.getItems()) {
        subtotal += i.getPrice() * i.getQuantity();
    }
    double tax = subtotal * 0.1;
    double total = subtotal + tax;
    System.out.println("Total: $" + total);
}
✅ After
void printOrder(Order order) {
    printHeader(order);
    printTotal(order);
}

private void printHeader(Order order) {
    System.out.println("Order #" + order.getId());
    System.out.println("Customer: " + order.getCustomer().getName());
}

private void printTotal(Order order) {
    Money total = calculateTotal(order);
    System.out.println("Total: " + total);
}

private Money calculateTotal(Order order) {
    Money subtotal = order.getItems().stream()
        .map(i -> i.getPrice().multiply(i.getQuantity()))
        .reduce(Money.ZERO, Money::add);
    return subtotal.addTax(TAX_RATE);
}
✓ Benefits
  • Self-documenting code
  • Enables reuse
  • Easier to test each piece
  • Reduces cognitive load
✗ Costs
  • More methods to navigate
  • Slight overhead from function calls (usually negligible)
Replace Conditional with Polymorphism

Move each branch of a conditional into an overriding method in a subclass or implementing class.

OCP PATTERN

Problem

You have a conditional that chooses different behavior depending on the type of an object. Every new type means updating every conditional everywhere.

❌ Type-switch anti-pattern
double calculateShipping(Order order) {
    switch (order.getShippingType()) {
        case STANDARD:
            return order.getWeight() * 2.5;
        case EXPRESS:
            return order.getWeight() * 5.0 + 10;
        case OVERNIGHT:
            return order.getWeight() * 8.0 + 25;
        case INTERNATIONAL:
            return order.getWeight() * 12.0 
                + order.getDestinationZone() * 15;
        default:
            throw new IllegalStateException();
    }
}
✅ Strategy Pattern
public interface ShippingStrategy {
    Money calculate(Order order);
}

@Component("STANDARD")
public class StandardShipping implements ShippingStrategy {
    public Money calculate(Order o) {
        return o.getWeight().times(2.5);
    }
}

@Component("EXPRESS")
public class ExpressShipping implements ShippingStrategy {
    public Money calculate(Order o) {
        return o.getWeight().times(5.0).plus(Money.of(10));
    }
}

// Adding DRONE delivery = add one class, zero changes
@Component("DRONE")
public class DroneShipping implements ShippingStrategy {
    public Money calculate(Order o) { ... }
}

@Service
public class ShippingCalculator {
    private final Map<String, ShippingStrategy> strategies;

    public Money calculate(Order order) {
        return strategies
            .getOrDefault(order.getShippingType(), 
                          standardShipping)
            .calculate(order);
    }
}
✓ Benefits
  • Open/Closed compliance
  • New types = zero existing changes
  • Each strategy independently testable
✗ Costs
  • More classes
  • May be overkill for 2-3 cases

Architectural Refactoring (HLD)

The Progression: Monolith → Microservices

Architectural refactoring is the highest-stakes form of refactoring. Done wrong, it can fragment a working system into a distributed monolith — all the operational complexity of microservices with none of the independence. The journey must be incremental and intentional.

Architecture Evolution Path

💡 Key Insight
Start with a Modular Monolith. Netflix, Shopify, and Stack Overflow ran as monoliths for years. The modular monolith is the often-overlooked middle ground — it gives you domain isolation without distributed systems complexity. Only extract services when you have clear operational or scaling reasons.

Before splitting services, enforce boundaries within your codebase. In Java/Spring, this means package-by-domain, not package-by-layer.

Hexagonal Architecture (Ports & Adapters)

The hexagonal architecture (Alistair Cockburn, 2005) isolates the application core from infrastructure concerns. Your domain logic has zero dependencies on Spring, JPA, or HTTP — it's pure Java. This makes it trivially testable and infrastructure-agnostic.

Refactoring in Spring Boot

Service Layer Refactoring

The service layer is the heart of a Spring Boot application. Common violations include services that reach directly into repositories of other domains, services that contain SQL logic, and services that return JPA entities directly to controllers.

❌ Anemic Service — all logic in controller
@RestController
public class UserController {
    @Autowired UserRepository repo;

    @PostMapping("/users")
    public User create(@RequestBody User user) {
        // Validation logic in controller
        if (user.getEmail() == null) throw new Exception();
        // Business logic in controller
        user.setCreatedAt(LocalDateTime.now());
        user.setStatus("ACTIVE");
        user.setPasswordHash(BCrypt.hash(user.getPassword()));
        return repo.save(user); // returns JPA entity!
    }
}
✅ Proper Layering
@RestController
@RequiredArgsConstructor
public class UserController {
    private final UserService userService;

    @PostMapping("/users")
    @ResponseStatus(HttpStatus.CREATED)
    public UserResponse create(
            @Valid @RequestBody CreateUserRequest req) {
        return userService.createUser(req);
    }
}

@Service
@RequiredArgsConstructor
public class UserService {
    private final UserRepository repo;
    private final PasswordEncoder passwordEncoder;
    private final UserMapper mapper;

    @Transactional
    public UserResponse createUser(CreateUserRequest req) {
        if (repo.existsByEmail(req.email())) {
            throw new DuplicateEmailException(req.email());
        }
        User user = User.create(req, passwordEncoder);
        return mapper.toResponse(repo.save(user));
    }
}

DTO vs Entity Separation

⚠ Critical Production Issue
Returning JPA entities from REST endpoints is one of the most common and dangerous mistakes in Spring Boot. It can expose internal fields (passwords, audit metadata), cause infinite recursion with bidirectional relationships, trigger unintended lazy loading, and lock your API to your database schema.
Proper DTO Layer with MapStruct
JAVA 
// Request DTO — what comes IN
public record CreateOrderRequest(
    @NotNull Long customerId,
    @NotEmpty @Valid List<OrderItemRequest> items,
    @NotNull ShippingAddressRequest shippingAddress
) {}

// Response DTO — what goes OUT (no JPA annotations, no cycles)
public record OrderResponse(
    UUID orderId,
    String status,
    MoneyDto total,
    List<OrderItemResponse> items,
    Instant createdAt
) {}

// MapStruct mapper — compile-time, zero reflection overhead
@Mapper(componentModel = "spring")
public interface OrderMapper {
    OrderResponse toResponse(Order order);
    Order toEntity(CreateOrderRequest req);

    @Mapping(target = "amount", source = "total.amount")
    @Mapping(target = "currency", source = "total.currency")
    MoneyDto toMoneyDto(Money money);
}

Solving the N+1 Query Problem

The N+1 problem is one of the most insidious performance bugs in JPA applications. You load N orders, then for each order JPA fires a separate query to load its items — N+1 total queries instead of 1.

❌ N+1 — Hibernate fires 101 queries for 100 orders
@Entity
public class Order {
    @OneToMany  // LAZY by default — good
    private List<OrderItem> items;
}

// Service
public List<OrderSummary> getRecentOrders() {
    List<Order> orders = repo.findAll(); // query 1
    return orders.stream().map(o -> {
        // EACH access fires a new query!
        int itemCount = o.getItems().size(); // query 2..N+1
        return new OrderSummary(o.getId(), itemCount);
    }).toList();
}
✅ Fetch Join — single query
// Option 1: JPQL fetch join
@Query("SELECT DISTINCT o FROM Order o " +
       "LEFT JOIN FETCH o.items " +
       "WHERE o.createdAt > :since")
List<Order> findRecentWithItems(
    @Param("since") Instant since);

// Option 2: @EntityGraph (declarative)
@EntityGraph(attributePaths = {"items", "items.product"})
List<Order> findAllByCustomerId(Long customerId);

// Option 3: Projection (best for read-only)
@Query("SELECT o.id as id, COUNT(i) as itemCount " +
       "FROM Order o LEFT JOIN o.items i " +
       "GROUP BY o.id")
List<OrderSummaryProjection> findOrderSummaries();

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(
            EntityNotFoundException ex) {
        return ErrorResponse.of("NOT_FOUND", ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(
            MethodArgumentNotValidException ex) {
        List<FieldError> errors = ex.getBindingResult()
            .getFieldErrors().stream()
            .map(f -> new FieldError(f.getField(), 
                                    f.getDefaultMessage()))
            .toList();
        return ErrorResponse.validation(errors);
    }
}

Testing During Refactoring

Tests are the safety net that makes refactoring possible. Without them, you're changing code while flying blind. The golden rule: don't refactor without tests; don't write tests after refactoring.

✅ The Refactoring Loop
Red → Green → Refactor. Write a failing test for the current behavior. Make it pass. Then refactor the implementation while keeping the test green. The test pins the observable behavior, freeing you to change internals freely.

            .register(registry);
    }

            try {
                OrderResponse response = orderService.createOrder(req);
                ordersCreated.increment();
                orderRevenue.record(response.total().amountInCents());
                return response;
            } catch (PaymentException e) {
                ordersFailedPayment.increment();
                throw e;
            }
        });

public class OrderProcessingRouter {

    private final LegacyOrderProcessor legacy;
    private final NewOrderService newService;

            request.getCustomerId()  // Percentage rollout by user
        );

            try {
                OrderResult result = newService.process(request);

                registry.counter("order.routing.fallback").increment();
                return legacy.process(request);
            }
        }

        registry.counter("order.routing", "service", "legacy").increment();
        return legacy.process(request);
    }
}

Real-World Case Study

FinTech Platform · 2023 · 18-month refactoring journey
From Monolith Meltdown to Modular Architecture

A Series B fintech company (payments processing, ~$2B TPV) was running a 6-year-old Spring MVC monolith with 180,000 lines of code. Deployment took 45 minutes. Any change to the payment module required regression testing the entire codebase. Incidents averaged 3 per week. Two senior engineers left citing codebase quality.

The Problem

  • Single PaymentProcessor class: 8,200 lines, 140 methods
  • 3 different database schemas shared across all domains via direct SQL
  • No domain boundaries — payment code directly calling user and fraud code
  • Test coverage: 12% (characterization tests: 0)
  • Average incident MTTR: 4.2 hours

The 4-Phase Approach

  1. Phase 1 (Months 1-3): Characterization tests. Wrote 340 tests to pin existing behavior. Coverage went from 12% → 41%. Zero feature work. Management was nervous.
  2. Phase 2 (Months 4-8): Internal modularization. Extracted 9 internal modules (Payment, Fraud, User, Ledger, etc.) with strict package boundaries. Enforced with ArchUnit. No external service splits.
  3. Phase 3 (Months 9-14): Strangler Fig for 3 highest-value services (Fraud Detection, Notification, Reporting). Used feature flags and shadow mode for 6 weeks before full cutover.
  4. Phase 4 (Months 15-18): Core Payment extraction. Most complex due to data ownership. Implemented Saga pattern for distributed transactions. Database split using dual-write with reconciliation.

Interview Questions

These questions are commonly asked for Senior → Staff-level engineering roles. Click to expand model answers.

Fundamentals

What is the difference between refactoring and rewriting? When would you choose each? +
Refactoring is incremental, behavior-preserving structural improvement with tests as a safety net. Rewriting discards the existing code and builds from scratch. Choose refactoring when the code is working but poorly structured (90% of cases). Choose rewriting only when: (1) the fundamental technology choice is wrong, (2) the domain model is so incorrect that incremental improvement would take longer than a rewrite, or (3) compliance/legal forces a greenfield. Rewrites routinely fail or cost 3-5x the estimate — they tend to lose accumulated edge-case handling. The Strangler Fig pattern bridges the gap: rewrite incrementally while keeping the system running.
How would you refactor a 5,000-line God class in a production system? +
First, don't touch it without tests. Step 1: Write characterization tests — 100-200 tests that pin the current behavior. Step 2: Identify responsibility clusters — group methods that operate on the same data or represent the same concern. Step 3: Use Extract Class to create candidate classes internally. Step 4: Introduce a thin facade (the original class name) that delegates to the new classes — this preserves the public API. Step 5: Verify all characterization tests still pass. Step 6: Gradually migrate callers to the new classes directly. Step 7: When the original class is empty delegation, delete it. Ship this over 4-8 weeks, not a single PR.
Explain the Strangler Fig pattern. When is it appropriate? +
The Strangler Fig pattern (Martin Fowler) routes new functionality to a new system while the legacy system handles existing behavior. You place an API gateway or facade in front of both systems and gradually move routes from legacy to new. It's appropriate when: migrating from monolith to microservices, replacing a legacy language/framework, or extracting a bounded context. The pattern requires a routing layer (can be API gateway, Nginx, or application-level router), feature flags to control traffic, shadow mode to validate the new system with production data before routing live traffic, and monitoring to detect behavioral divergence.
How do you handle the N+1 query problem in Spring Data JPA? +
The N+1 problem occurs when fetching N entities triggers N additional queries for associations. Detection: enable p6spy or datasource-proxy logging; Hibernate's statistics mode. Solutions in order of preference: (1) JPQL fetch join for specific query needs — `SELECT o FROM Order o JOIN FETCH o.items WHERE...`, (2) @EntityGraph for declarative fetching without JPQL, (3) Projections (interface or record) for read-only use cases — pull only needed columns in one query, (4) Batch fetching via `@BatchSize` on collections — Hibernate batches N queries into ceil(N/size) queries. Avoid eager fetching globally — it's a shotgun solution that over-fetches. The right answer depends on the use case: reporting queries → projections, operational queries → fetch joins.
You're asked to migrate a monolith to microservices. What's your approach? +
I'd push back on the framing first — "microservices" is rarely the right starting point. My approach: (1) Modular Monolith First: enforce package-by-domain boundaries, forbid cross-domain repository access, identify bounded contexts. This alone solves 60% of problems. (2) Identify extraction candidates: services with independent scaling needs, different release cycles, or clear ownership boundaries. Don't extract just to extract. (3) Use the Strangler Fig: route at the API gateway, shadow test the new service, incrementally move traffic with feature flags. (4) Data: the hardest part. Database-per-service requires saga patterns for transactions. Consider dual-write with reconciliation during transition. (5) Operational readiness: distributed tracing, centralized logging, service mesh before extracting services. Microservices without observability are an operational nightmare.
How do you measure the success of a refactoring effort? +
Engineering metrics: test coverage (before/after), cyclomatic complexity (SonarQube), code duplication percentage, build time, deployment frequency. Operational metrics: incident rate (MTTF), MTTR, change failure rate. Business/team metrics: feature velocity (story points per sprint), time to onboard a new engineer, developer satisfaction (team surveys). The most convincing metric for leadership is usually feature velocity — "we shipped 3x as many features in Q3 after the refactoring as in Q2." Track all metrics before, during, and after to tell a complete story.
What is the Hexagonal Architecture and why is it useful for refactoring? +
Hexagonal Architecture (Ports and Adapters, Alistair Cockburn 2005) separates the application core from all infrastructure. The core (domain logic) depends only on abstractions (ports) — interfaces like OrderRepository, PaymentGateway. Adapters implement these ports: JpaOrderRepository, StripePaymentGateway. The core has zero imports from Spring, JPA, or HTTP. Benefits for refactoring: (1) Domain logic is testable with pure unit tests — no Spring context, no database, (2) You can swap infrastructure (MySQL → Postgres, REST → messaging) without touching domain logic, (3) Makes bounded context boundaries explicit and testable with ArchUnit rules. In Spring Boot, this means the domain package imports nothing from org.springframework, and Spring beans live only in the infrastructure/adapter layer.

Behavioral / Leadership

How do you convince stakeholders to allocate time for refactoring? +
Frame it as risk and cost, not code quality. Show: (1) Incident cost — "Our last 3 incidents took an average of 4 hours to resolve and cost $X in customer SLA credits. The root cause was in the OrderManager class every time." (2) Velocity data — "Features that touch the payment module take 3x longer than features in newer, cleaner code." (3) Engineer retention — "Two engineers explicitly cited codebase quality in exit interviews." Don't ask for a "refactoring sprint" — ask to embed refactoring into feature work (20% rule). Show quick wins early. Track and report metrics every sprint. Never frame it as "cleaning up" — always connect it to a business outcome.

The interviewer is assessing: systems thinking, communication, risk awareness, outcome orientation.
